<summary><strong>v2.0.0</strong> — Replication 등록 API 단순화 및 GET 엔드포인트 Center 필터 추가</summary>

**Breaking Changes — Replication 등록 Body Schema 단순화**

- `POST /replications` — `sourceRepository`, `targetRepository`, `sourceServer` 필드를 단순 문자열(comma-separated)로 변경하고, `backupJobName` 필드를 `backup`으로 통합
  - `sourceRepository`: 배열 → 문자열 (예: `"39,40"`)
  - `targetRepository`: 객체 → 문자열 (예: `"41"`)
  - `sourceServer`: 배열 → 문자열 (예: `"1,server1"`)
  - `backupJobName`(배열) → `backup`(문자열, 예: `"1,job1,123"`)
- v1 호환 엔드포인트(`POST /replications`, `REPLICATION_VERSION=1` 환경)도 `sourceRepository`/`targetRepository`를 문자열로 변경

**신규 기능 — GET 엔드포인트 Center 필터 추가**

- 모든 조회 엔드포인트에 `center` query parameter 지원 (comma-separated, ID/Name 혼용 가능)
- 적용 엔드포인트: `/backups`, `/backups/histories`, `/recoveries`, `/recoveries/histories`, `/replications`, `/replications/histories`, `/os-replications`, `/os-replications/histories`, `/servers`, `/servers/:identifier/partitions`, `/schedules`, `/licenses`, `/cloud-auth/zos`, `/cloud-auth/recovery`, `/zdms/repositories`
- 예: `?center=destconm,destconm2` 또는 `?center=9,10`

**Bug Fix — Backup Image 조회 정확성 개선**

- `POST /recoveries` — `backupFile` 미지정 케이스에서 사용자가 지정한 Center가 아닌 다른 Center에서 backup image를 조회하던 문제 수정
  - 영향 시나리오: 여러 Center를 운영하는 환경에서 잘못된 (Center, Repository) 조합 시도로 인해 `Backup image does not exist` 에러가 발생하던 케이스
- `POST /recoveries`, `GET /backups/images/server/:serverName` — 다른 Center/User/Repository 조합의 동일 파일명 데이터가 잘못 매칭되던 문제 수정 (Center/User/Repository 조건 함께 검증하여 정확한 결과 보장)

**운영 개선**

- Graceful Shutdown 타임아웃 30초 → 5초 단축, 종료 시 연결 즉시 정리
- Replication 버전 분기 처리를 런타임 일관성을 갖도록 통합

**Bug Fix — Schedule 처리 정책 정리**

- `PUT /backups/:identifier` — schedule 변경 시 동작 규칙 명문화
  - 모드 동일(F/I→F/I, Smart→Smart) + 기존 스케줄 있음 → 기존 스케줄의 데이터만 입력값으로 갱신 (스케줄 ID 보존)
  - 모드 전환(F/I↔Smart) → 신규 스케줄 등록 (기존 스케줄 참조 해제)
  - 기존 스케줄 없음 → 신규 스케줄 등록
  - 입력이 스케줄 ID(숫자) + 기존 스케줄 있음 → 입력 ID의 스케줄 데이터를 기존 스케줄에 복사하여 갱신 (스케줄 ID 보존)
- 기존 스케줄 ID 참조 시 충돌이 발생하던 문제 해결

**Bug Fix — Schedule 작업명 매핑**

- `POST /recoveries`, `PUT /recoveries/:identifier`, `PUT /backups/:identifier` — 스케줄 등록/변경 시 작업 이름이 스케줄에 매핑되지 않던 문제 수정
- `POST /recoveries` 호출 흐름에서 스케줄 등록 시점을 작업 이름 확정 이후로 이동하여 정상 매핑 보장

**운영 개선 — Recovery 등록 Backup Image 조회 Timeout 동적화**

- `POST /recoveries` — backup image 조회 timeout이 처리 대상 파티션 수에 비례하여 자동 확장 (파티션당 30초)
- 다중 파티션을 한 번에 처리할 때 고정 timeout으로 인해 실패하던 문제 해결
- 모든 파티션이 timeout 전 완료되면 즉시 종료

**UX 개선 — 에러 응답 메시지 구체화**

- 모든 엔드포인트에서 필수 필드 누락 시 응답 메시지가 `"Invalid input"` → **`"<field> is required"`** 로 자동 치환
- `Internal server error.` 단독 응답이 사라지고, 원본 에러 메시지가 함께 포함됨 (예: `"Internal server error: <원본 메시지>"`)
- `PUT /backups/:identifier`의 에러 응답에 어느 단계에서 어떤 원인으로 실패했는지 detail 자동 포함
- Smart schedule 호환성 검증 실패 메시지에 실제 값 포함 (예: `"Basic schedule type(7)과 Advanced schedule type(8)이 일치하지 않습니다."`)

**License 도메인 정합화 (Breaking Change 일부 포함)**

- `POST /licenses` 응답 — `copies.usage` 타입이 `string("0%")` → **`number(0)`** 으로 변경, `dates.daysRemaining` 타입이 `string("expired"|"N")` → **`number(만료=0)`** 으로 변경 (Get 응답과 형식 통일)
- `POST /licenses` / `PUT /licenses/assign` 요청 본문의 `user` 필드 — 두 엔드포인트 모두 `string | number` 모두 허용으로 통일 (기존: assign은 number-only, regist는 string-only로 분화되어 있었음). 사용자 ID(숫자) / 이메일·이름(문자열) 모두 입력 가능
- `PUT /licenses/assign` 요청 본문의 `center` — 빈 문자열 입력 명시적 거부 (기존: 통과 후 service 분기에서 무시)
- `GET /licenses` 응답 — `copies.available`이 비정상 데이터(used > total)에서 음수로 노출되던 것을 `Math.max(0, ...)`로 가드
- `PUT /licenses/assign` — 트랜잭션 진입 전 사전 검증 3종 추가 (Behavior Change)
  - License와 Server가 동일 center에 속하는지 확인 (불일치 시 **403 CENTER_MISMATCH**)
  - 대상 Server에 이미 라이선스가 할당되어 있으면 거부 (**409 ALREADY_ASSIGNED**)
  - `license_history`에 `(sSystemName, nCenterID)` 매칭 row가 존재하면 거부 — 한 번 할당된 server는 재할당 불가 (**409 ALREADY_ASSIGNED**)
  - 모든 검증이 transaction 진입 전에 실행되어 실패 시 license 검증 disable/enable 사이클이 일어나지 않음
